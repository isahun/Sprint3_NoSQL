Introducció

En aquesta activitat de l'Sprint 3 he desplegat una base de dades al núvol fent servir MongoDB Atlas. He reutilitzat el model de l'activitat anterior per crear una col·lecció d'articles representada com un array d'objectes en format JSON i d'aquesta manera importar dades de prova a la base de dades.

Objectius

Importar un fitxer JSON amb dades d'exemple

Verificar que les dades s'han importat i es consulten correctament

1. Creació del Clúster

El primer pas és crear un clúster gratuït (nivell M0) seleccionant el proveïdor de núvol de la regió més propera. En aquest cas, el proveïdor és AWS/Paris.

2. Creació d'Usuari

S'ha creat un usuari de base de dades amb credencials específiques per accedir a les col·leccions de manera segura.

3. Configuració d'Accés per IP

En aquest punt s'ha decidit configurar l'accés de xarxa de manera que es permetin connexions des de qualsevol IP, a través de la IP 0.0.0.0/0. Aquesta és una decisió excepcional fruit de l'entorn acadèmic en què es desenvolupa la tasca.

4. Connexió amb MongoDB Compass

MongoDB Compass és una interfície gràfica que permet administrar bases de dades MongoDB. La connexió amb el clúster desplegat al núvol s'ha realitzat mitjançant la connection string generada per MongoDB Atlas.

5. Importació de Dades

S'ha importat un arxiu JSON per poblar la base de dades amb la informació de l'activitat. El JSON conté un array d'objectes, en concret 3 articles, amb la mateixa informació que l'exercici anterior: títol, contingut, data de publicació, autor i comentaris.

7. Anàlisi del Model NoSQL

S'ha creat la col·lecció articles a MongoDB, on cada document representa un article amb la següent estructura:

  • • titol
  • • contingut
  • • data_publicacio
  • • autor (objecte embegut amb id i email)
  • • comentaris (array d'objectes amb contingut, data i autor)

Avantatges

  • • Tota la informació d'un article es troba en un únic document.
  • • Recuperació ràpida de dades amb una sola consulta.
  • • No cal gestionar múltiples col·leccions relacionades.
  • • El format coincideix amb l'estructura JSON utilitzada al frontend.

Inconvenients

  • • Redundància de dades (l'autor es repeteix en diversos documents).
  • • Si un usuari canvia dades, cal actualitzar múltiples documents.
  • • L'acumulació de comentaris pot fer créixer excessivament la mida del document.

Conclusions

La base de dades s'ha desplegat correctament en un entorn DBaaS fent servir MongoDB Atlas. S'ha comprovat la connexió remota, la correcta importació de les dades i les relacions establertes entre elles. Pel que fa el model emprat, en el context d'un blog acadèmic amb moltes lectures i una estructura estable, el model embegut és una solució eficient i coherent.
